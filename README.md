# Git 协作规范（4人小团队·零冲突版）
> **核心目标：用最简单的流程，避免99%的代码冲突**
> 
> 本规范专为4人小团队设计，去掉所有不必要的复杂流程，只保留防冲突的核心规则。所有人严格遵守，基本不会出现大冲突。

---

## 🚨 防冲突黄金法则（必须背下来）
1. **永远不要直接在 `main` 分支上写代码**
2. **写代码前先拉，提交前再拉**（一天至少拉2次）
3. **分支要小，提交要勤**（一个分支只做一件事，每天至少提交1次）
4. **谁改了公共文件，群里说一声**

---

## 一、分支管理（单主分支模型，最简单）
我们只保留一个长期分支 `main`，所有开发都在临时分支上进行。

### 分支命名规范（简单好记）
```
类型/你的昵称-做什么事
```

| 类型 | 说明 | 示例 |
|------|------|------|
| `feature` | 开发新功能 | `feature/小明-产品列表页` |
| `bugfix` | 修复bug | `bugfix/小红-图片加载失败` |
| `hotfix` | 紧急修复线上问题 | `hotfix/小刚-登录报错` |

### 分支生命周期
1. 从最新的 `main` 分支创建你的开发分支
2. 开发完成后，提交PR合并到 `main`
3. 合并后立即删除本地和远程的开发分支

---

## 二、提交信息规范（只保留最常用的）
### 标准格式
```
<type>: 一句话说清楚做了什么
```

### 只用这6个type足够
| type | 说明 |
|------|------|
| `feat` | 新增功能 |
| `fix` | 修复bug |
| `docs` | 改文档 |
| `style` | 改格式（缩进、空格等，不影响运行） |
| `refactor` | 代码重构（没加功能也没修bug） |
| `chore` | 改配置、装依赖等杂事 |

### 好的示例 vs 坏的示例
✅ 好：`git commit -m "feat: 新增产品搜索功能"`  
❌ 坏：`git commit -m "更新"`  
❌ 坏：`git commit -m "今天写了很多代码"`

## 三、每日开发流程（一步都不能少）
### 1. 早上开工第一步（必做）
```bash
# 切换到main分支
git checkout main

# 拉取所有人昨晚提交的最新代码
git pull origin main
```

### 2. 创建自己的开发分支（必做）
```bash
# 创建并切换到新分支
git checkout -b feature/你的昵称-功能名
```

### 3. 开发过程中（每2-3小时做一次）
```bash
# 查看自己改了哪些文件
git status

# 添加要提交的文件（推荐单个加，不要用 git add .）
git add src/pages/Product.jsx
git add src/components/SearchBar.jsx

# 提交代码
git commit -m "feat: 完成搜索框基础功能"
```

### 4. 准备推送前（最关键的防冲突步骤！必做）
```bash
# 再次拉取main分支的最新代码，合并到自己的分支
git pull origin main
```

- ✅ 如果没有冲突：直接继续下一步
- ⚠️ 如果有冲突：
  1. Git会提示哪些文件有冲突，打开文件
  2. 找到 `<<<<<<< HEAD` 和 `>>>>>>> main` 之间的部分
  3. 和写这段代码的同学沟通，保留双方需要的代码
  4. 解决完所有冲突后，执行：
     ```bash
     git add .
     git commit -m "merge: 解决与main分支的冲突"
     ```

### 5. 推送分支到远程
```bash
git push origin feature/你的昵称-功能名
```

### 6. 创建PR并合并
1. 打开GitHub仓库，点击自动出现的 **Compare & pull request**
2. 标题和提交信息一样即可，描述简单写一下做了什么
3. 点击 **Create pull request**
4. 喊一个同学帮你看一眼代码（1分钟就行）
5. 没问题就点击 **Merge pull request** → **Confirm merge**
6. 点击 **Delete branch** 删除远程分支

### 7. 本地清理（必做，否则分支会越来越多）
```bash
# 切换回main分支
git checkout main

# 拉取合并后的最新代码
git pull origin main

# 删除本地已经合并的分支
git branch -d feature/你的昵称-功能名
```

## 四、绝对禁止的操作（踩了必出冲突）
1. ❌ **绝对禁止直接推送到main分支**
2. ❌ **绝对禁止一个分支写超过3天**（功能拆细，一个分支只做一件事）
3. ❌ **绝对禁止一次提交超过10个文件**（拆成多个小提交）
4. ❌ **绝对禁止强制推送（git push -f）到main分支**
5. ❌ **绝对不要在本地攒一周的代码再提交**
6. ❌ **绝对不要修改别人正在写的文件，如果必须改，先在群里说一声**

---

## 五、冲突解决指南
### 1. 小冲突（自己能解决的）
按照上面第4步的方法，在本地解决后再提交。

### 2. 大冲突（看不懂、不敢改的）
**不要硬改！** 直接在群里@写这段代码的同学，两个人一起解决。

### 3. 冲突太多，完全搞砸了
```bash
# 放弃本地所有修改，重新拉取最新代码
git reset --hard origin/main

# 然后重新创建分支，把自己的代码一点点复制进去
```

## 七、常用命令速查
```bash
# 查看当前分支状态
git status

# 查看所有分支
git branch

# 切换分支
git checkout 分支名

# 查看提交历史
git log --oneline

# 撤销最后一次提交（保留代码）
git reset --soft HEAD~1

# 放弃本地所有修改
git checkout .
```
